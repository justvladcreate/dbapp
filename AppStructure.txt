Описание приложения

Представим, что я являюсь бухгалтером или сотрудником организации, занимающейся сдачей недвижимости в аренду.

В процессе работы в организацию регулярно поступают новые документы: сведения о собственниках, арендаторах, объектах недвижимости и договорах аренды. Вся эта информация должна храниться в базе данных и поддерживаться в актуальном состоянии.

Для упрощения работы было разработано приложение для управления базой данных учёта аренды недвижимости.

Приложение должно позволять:

просматривать данные, содержащиеся в базе данных;
выполнять поиск, сортировку и фильтрацию данных;
добавлять новые записи;
редактировать существующие записи;
удалять записи с соблюдением правил целостности данных;
получать сводную информацию об аренде недвижимости;
экспортировать данные в CSV-файлы.

После запуска приложения пользователь должен видеть главную страницу системы.

Навигация между разделами осуществляется через верхнее меню.

Основные разделы приложения:

Главная;
Договоры;
Собственники;
Арендаторы;
Объекты недвижимости.
Общие требования к интерфейсу

Для всех разделов приложения должны соблюдаться следующие требования:

Все действия пользователя должны выполняться максимально быстро и сразу отображаться в интерфейсе.
Поиск и фильтрация должны работать в режиме реального времени без необходимости нажимать дополнительные кнопки.
Все таблицы должны поддерживать сортировку по возрастанию и убыванию.
Пользователь не должен видеть внутренние идентификаторы (ID), используемые базой данных для хранения связей между таблицами.
При удалении записи должны выводиться предупреждения о последствиях удаления.
Пользователь должен редактировать только данные текущей сущности.

Например:

адрес объекта недвижимости редактируется только на странице «Объекты недвижимости»;
данные арендатора редактируются только на странице «Арендаторы»;
данные собственника редактируются только на странице «Собственники»;
данные договора редактируются только на странице «Договоры».

Такой подход исключает случайное изменение связанных данных.

Главная страница

Главная страница предназначена для просмотра сводного отчёта по аренде недвижимости.

На странице отображается полная таблица данных, сформированная на основе представления (VIEW), объединяющего сведения о:

договорах;
арендаторах;
собственниках;
объектах недвижимости;
параметрах аренды.

Как пользователь я хочу:

Видеть всю информацию по аренде в одном месте.
Выполнять поиск по любому отображаемому полю.
Сортировать данные по любому столбцу.
Экспортировать текущий результат отображения в CSV-файл.
Получать результат поиска и фильтрации сразу после ввода данных.
Иметь возможность открыть карточку редактирования любой записи.

При этом редактирование на главной странице напрямую запрещено.

Если пользователь выбирает редактирование какого-либо элемента, приложение должно автоматически открыть соответствующий раздел и сразу показать форму редактирования выбранной записи.

Например:

редактирование договора открывает раздел «Договоры»;
редактирование арендатора открывает раздел «Арендаторы»;
редактирование собственника открывает раздел «Собственники»;
редактирование объекта открывает раздел «Объекты недвижимости».

Добавление новых записей на главной странице запрещено.

Раздел «Договоры»

Раздел предназначен для управления договорами аренды.

Здесь пользователь может просматривать, добавлять, изменять и удалять договоры.

При создании нового договора обязательно должны быть заполнены следующие поля:

дата заключения договора;
арендатор;
собственник;
объект недвижимости;
цена аренды.

Один договор может содержать несколько объектов недвижимости.

Для каждого объекта внутри договора может быть указана собственная:

цена аренды;
дата начала аренды;
срок аренды.

Поскольку договор заключается между одним арендатором и одним собственником, при добавлении дополнительного объекта все данные арендатора и собственника должны автоматически переноситься в новую запись.

Если дата начала аренды не указана, система должна:

предупредить пользователя;
автоматически использовать дату заключения договора.

Как пользователь я хочу:

Сортировать договоры по любому полю.
Искать договоры по любому полю.
Использовать несколько фильтров одновременно.
Чтобы фильтрация работала по принципу AND.
Не видеть технические идентификаторы базы данных.
Редактировать существующие договоры.
После добавления договора сразу видеть его в отдельном информационном блоке в верхней части страницы.
Иметь кнопку «Показать», которая автоматически прокручивает страницу к добавленному договору.
Раздел «Собственники»

Раздел предназначен для работы с собственниками недвижимости.

Как пользователь я хочу:

Просматривать список всех собственников.
Добавлять новых собственников.
Изменять данные существующих собственников.
Удалять собственников.
Выполнять поиск и сортировку по всем полям.
Видеть список принадлежащих собственнику объектов недвижимости.
Изменять принадлежность объектов недвижимости.

Один объект недвижимости может принадлежать только одному собственнику.

Также я хочу видеть:

в каких договорах участвовал собственник;
краткую статистику по договорам;
возможность открыть подробную информацию по каждому договору.

При удалении собственника должно выводиться предупреждение о последствиях.

Если удаление подтверждено, должны быть удалены:

все связанные договоры;
все записи учёта аренды, связанные с этими договорами.

После удаления ни один договор не должен содержать ссылок на несуществующего собственника.

Раздел «Арендаторы»

Раздел предназначен для управления арендаторами.

Функциональность аналогична разделу «Собственники».

Как пользователь я хочу:

Просматривать список арендаторов.
Добавлять новых арендаторов.
Изменять данные арендаторов.
Удалять арендаторов.
Искать и сортировать данные.
Видеть все договоры, в которых участвует выбранный арендатор.
Получать краткую сводку по договорам арендатора.

В отличие от собственников арендаторы не могут владеть объектами недвижимости.

При удалении арендатора должны удаляться все связанные договоры и записи учёта аренды.

Раздел «Объекты недвижимости»

Раздел предназначен для управления объектами недвижимости.

Как пользователь я хочу:

Просматривать список всех объектов.
Добавлять новые объекты.
Редактировать существующие объекты.
Удалять объекты.
Выполнять поиск и сортировку по всем полям.
Видеть собственника каждого объекта.
Видеть список договоров, в которых участвует объект.
Открывать подробную информацию по связанным договорам.

При удалении объекта должны удаляться все записи учёта аренды, связанные с этим объектом.

Если после удаления объекта договор становится недействительным и больше не содержит ни одного объекта недвижимости, такой договор также должен быть удалён из системы.

Правила целостности данных

Система должна обеспечивать следующие правила:

Договор не может существовать без арендатора.
Договор не может существовать без собственника.
Договор не может существовать без объекта недвижимости.
Один объект недвижимости может принадлежать только одному собственнику.
Один договор может включать несколько объектов недвижимости.
Для каждого объекта внутри договора могут быть заданы собственные условия аренды.
Если дата начала аренды не указана, автоматически используется дата заключения договора.
После удаления любой связанной сущности в системе не должны оставаться невалидные договоры или записи учёта аренды.

Дополнительные требования к пользовательскому интерфейсу
Обратная связь интерфейса
Любое действие пользователя должно сопровождаться немедленной визуальной реакцией интерфейса.
После добавления, изменения или удаления данных все связанные страницы и таблицы должны автоматически обновляться без необходимости перезапуска приложения.
Пользователь должен иметь возможность вручную обновить данные на любой странице при помощи кнопки «Обновить».
При выполнении длительных операций должна отображаться индикация загрузки:
индикатор прогресса;
сообщение о загрузке;
либо другой визуальный элемент, показывающий, что приложение продолжает работу.
Требования к вводу данных
Проверка данных до сохранения
Все данные должны проверяться непосредственно во время ввода пользователем.
Пользователь не должен иметь возможность сохранить запись с некорректными данными.
При обнаружении ошибки диалоговое окно не должно закрываться до её исправления.
Все сообщения об ошибках должны содержать понятное описание проблемы и способ её исправления.
Числовые поля
Для числовых данных должны использоваться специализированные элементы интерфейса:
QSpinBox;
QDoubleSpinBox;
либо поля с валидаторами.
Пользователь не должен иметь возможность вводить буквы или специальные символы в числовые поля.
Для каждого числового поля должны быть установлены допустимые минимальные и максимальные значения.
Приложение не должно завершаться с ошибкой из-за некорректного пользовательского ввода.
Паспортные данные
Поле паспорта должно проверяться непосредственно в форме ввода.
До сохранения должны контролироваться:
длина значения;
допустимые символы;
обязательность заполнения.
При ошибке пользователь должен иметь возможность исправить данные без повторного открытия формы.
Подсказки пользователю
Все поля ввода должны содержать всплывающие подсказки.
Подсказка должна объяснять:
назначение поля;
допустимый формат ввода;
ограничения по значению.
Требования к работе с договорами
Добавление договора
При выборе собственника список доступных объектов недвижимости должен автоматически обновляться во всех элементах интерфейса текущего договора.
Пользователь не должен иметь возможность выбрать объект, принадлежащий другому собственнику.
Если у собственника отсутствуют объекты недвижимости:
пользователь должен получить уведомление сразу после выбора собственника;
кнопка добавления объекта должна быть недоступна;
должно отображаться пояснение о необходимости сначала добавить объект недвижимости.
Работа с датой начала аренды
Пользователь должен иметь возможность явно указать, что дата начала аренды отсутствует.
Интерфейс должен наглядно показывать, что в таком случае будет использована дата договора.
Не допускается появление дополнительных подтверждений, если поведение уже было выбрано пользователем через соответствующий элемент интерфейса.
Поведение интерфейса должно быть очевидным и не создавать противоречий между выбранным состоянием элемента управления и фактическим результатом сохранения.
Отображение сумм
Денежные значения должны отображаться в формате, удобном для чтения.
Для денежных сумм должны использоваться:
разделители тысяч;
обозначение валюты.

Пример:

1 250 000 ₽
Требования к фильтрации и поиску
Все страницы со списками данных должны иметь кнопку «Сбросить фильтры».
После сброса пользователь должен видеть полный набор записей.
Значения фильтров по умолчанию не должны скрывать существующие данные.
Пустое поле фильтра должно означать отсутствие ограничений.
Пользователь должен явно видеть все активные фильтры.
Требования к удалению данных
Предупреждения об удалении
Перед удалением система должна отображать полный список затрагиваемых объектов.
Пользователь должен видеть:
какие договоры будут удалены;
какие объекты будут удалены;
какие записи учёта аренды будут удалены.
Недостаточно отображать только количество затронутых записей.
Каскадное удаление
При удалении собственника должны удаляться:
связанные объекты недвижимости;
связанные договоры;
связанные записи учёта аренды.
При удалении арендатора должны удаляться:
связанные договоры;
связанные записи учёта аренды.
При удалении объекта недвижимости должны удаляться:
связанные записи учёта аренды.
Если после удаления объекта договор больше не содержит объектов недвижимости, такой договор также должен быть удалён.
Перед подтверждением удаления пользователь должен получать подробное предупреждение о последствиях операции.
Требования к редактированию данных
При наличии несохранённых изменений пользователь должен получать предупреждение перед закрытием формы.
Пользователь должен иметь возможность:
сохранить изменения;
отказаться от изменений;
вернуться к редактированию.
Все диалоги должны поддерживать горячие клавиши:
Enter — подтвердить действие;
Esc — отменить действие;
Ctrl+S — сохранить изменения.
Требования к главной странице
Главная страница должна автоматически обновляться после любых изменений данных в системе.
Пользователь должен иметь возможность вручную обновить данные.
Для строк таблицы должно поддерживаться контекстное меню по нажатию правой кнопки мыши.
Контекстное меню должно предоставлять быстрый доступ к операциям:
редактирование договора;
редактирование арендатора;
редактирование собственника;
редактирование объекта недвижимости.
Информационный блок о последнем добавленном договоре должен автоматически скрываться:
после обновления страницы;
после перехода на другую страницу;
после закрытия приложения.
